## Overlap & precedence notes

### `tdd` vs `test-driven-development`

Both are red-green-refactor TDD skills with no directory-level redirect between them. `tdd` frames the discipline around vertical-slice "tracer bullets" (one test → one implementation → repeat); `test-driven-development` states the same Iron Law (no production code without a failing test first) more tersely. Until they're merged, either satisfies "write a failing test first" guidance; prefer `test-driven-development` as the canonical name referenced elsewhere in this catalog and in org-level instructions.

### `repository-organization` vs `nix-flake-organization`

`repository-organization` covers general repo layout, agent context, docs, scripts, skills, presets, and ADR placement. `nix-flake-organization` is the deeper supplement for Nix flake-specific output structure: thin `flake/` public layer and `src/` implementation. Apply `repository-organization` broadly; use `nix-flake-organization` when the task is specifically about flake layout.

### `grill-me` vs `grill-with-docs`

`grill-me` is a pure Q&A skill — relentless questioning through the decision tree, no side effects. `grill-with-docs` is the version that also updates `CONTEXT.md` and creates ADRs inline as decisions crystallize. Use `grill-me` when you just want to stress-test a plan; use `grill-with-docs` when you also want durable documentation of the decisions reached.

## Known gaps — referenced by name but not shipped as skills

The following names are referenced by real skills below (e.g. `brainstorming` names `writing-plans` as its mandatory next step; `using-superpowers` and `brainstorming` name `frontend-design`; `ui-component-architecture` and `vercel-react-best-practices` name `coding-standards`) or by earlier versions of this catalog, but **no `skills/<name>/` directory exists for them**. Invoking them by name will not resolve to an installed skill. `coding-standards` and `hl-funding-analysis` were deliberately removed (see `5347d8d`); the rest have no history in this repo and appear to have been aspirational catalog entries that were never built:

`browser-use`, `caveman`, `caveman-review`, `coding-standards`, `compress`, `dispatching-parallel-agents`, `executing-plans`, `frontend-design`, `hl-funding-analysis`, `neon-postgres`, `subagent-driven-development`, `systematic-debugging`, `verification-before-completion`, `writing-plans`

Before pointing a new org-level instruction file at one of these, either build the skill via `dm-skill-creator` or update the referencing skill body to name a real skill instead.

## Client/runtime policy docs

`using-superpowers` and `continuous-learning` are **not ordinary task skills** that implement hooks or produce artifacts themselves. They are **runtime policy documents** consumed by the agent client (e.g., OpenCode, Claude Code) to configure session behavior:

- **`using-superpowers`**: Instructs the agent runtime to always check for and invoke relevant skills before responding. It is a _dispatch policy_, not a task implementation.
- **`continuous-learning`**: Instructs the agent runtime to extract reusable patterns at session end. It is a _learning policy_, not a task implementation.

`strategic-compact` similarly configures auto-compaction behavior for long autonomous sessions.

## How to add an entry

When you add a skill at `skills/<name>/`:

1. Pick a one-line description that overlaps with `SKILL.md` frontmatter `description` but is human-skim-friendly.
2. Add a row to the table above.
3. If the skill overlaps with an existing one, add a precedence note in the section above.
4. If the skill ships scripts that depend on environment (Python, Node, particular CLIs), note that in a "Notes" column or in a footnote.
